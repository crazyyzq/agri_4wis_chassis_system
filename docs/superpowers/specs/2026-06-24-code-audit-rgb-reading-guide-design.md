# ECU 测试固件代码审计、RGB 状态灯与阅读指南设计

日期：2026-06-24

## 目标

1. 对 `ecu_board_test` 应用、`ecu_isolation` 板级支持包和构建脚本进行完整静态审计。
2. 修复审计中能够复现的缺陷，不以“增加注释”为由改变既有测试判定逻辑。
3. 为公共接口、模块职责、硬件假设、安全边界和不直观算法补充有效注释。
4. 增加独立 RGB 状态灯驱动，使无串口时也能判断固件所处阶段。
5. 编写按推荐顺序组织的项目结构和逐文件阅读指南。

## 注释原则

- 每个手写 `.c/.h` 文件包含模块职责说明。
- 公共函数声明说明输入、输出、索引规则、阻塞性和副作用。
- 重点注释硬件极性、单位、超时、安全清理、状态转换及算法原因。
- 不给自动生成的 `pinmux.c/.h/.hpmpc` 逐行加注释；保留工具生成内容，阅读指南解释其来源。
- 不写重复代码字面含义的行尾注释，避免注释噪声和后续失真。

## RGB 状态灯设计

新增应用层 `status_led` 模块，板级 `board_rgb_write()` 仍只负责低电平有效 GPIO。

状态定义：

- `STATUS_LED_BOOTING`：红灯常亮，表示已进入 `main()`、尚未完成自测。
- `STATUS_LED_READY`：绿灯约 1 Hz 心跳，表示 CLI 可用、主循环存活。
- `STATUS_LED_TESTING`：蓝灯快速闪烁，表示测试用例正在执行。
- `STATUS_LED_FAILED`：红灯快速闪烁，表示自测或板级测试失败。

驱动采用非阻塞 `status_led_poll()`；应用循环和测试中的可中止等待点负责调用。驱动不创建中断，不占用额外定时器，以免影响通信测试。

`RGB.ALL` 通过 `status_led_override_begin()` 暂停状态动画并独占三色灯；cleanup 必须调用 `status_led_override_end()`，恢复测试前状态。底层始终采用先全灭、后点亮的顺序，避免短暂混色。

## 集成点

- `main.c`：`board_init()` 后进入 BOOTING；自测失败进入 FAILED；成功后进入 READY。
- `app_shell.c`：等待命令时轮询心跳；运行测试前进入 TESTING；根据结果恢复 READY 或进入 FAILED。
- `test_runner.c`：长测试轮询 abort 时同时推进状态灯动画；所有退出路径仍执行 Safety Manager 清理。
- `test_rgb.c`：测试期间取得 RGB 独占权，cleanup 释放。

RGB 状态不参与 Safety Manager 的危险资源模型，因为 LED 不会驱动车辆执行机构；但所有状态切换仍保证索引和极性合法。

## 审计与验证

审计范围包括：边界检查、空指针、数组长度、阻塞循环、整数溢出、硬件极性、错误路径 cleanup、Flash/EEPROM 破坏范围、CLI/JSON 边界、CAN/UART/SBUS 帧解析、GNU/SES 构建一致性。

验证顺序：

1. 为 RGB 状态转换和独占/恢复编写目标侧失败自测。
2. 实现状态灯模块并使自测编译通过。
3. 运行 GNU `flash_sdram_xip` 全量构建。
4. 重新生成合法 SES 工程并用 SES 8.28 `emBuild` 编译 Debug。
5. 检查链接保留区 `0x807F0000..0x80800000`。
6. 下载到已连接板卡，观察启动和运行状态；无串口时只确认灯态与 J-Link PC，不将硬件功能测试标为 PASS。
